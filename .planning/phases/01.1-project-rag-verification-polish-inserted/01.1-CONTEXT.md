# Phase 1.1: Project RAG — Verification & Polish - Context

**Gathered:** 2026-04-22
**Status:** Ready for planning

<domain>
## Phase Boundary

The Project RAG feature (per-project document library with semantic retrieval that enriches live answers and powers an Ask-this-project flow) is already implemented on `main` across commits `cf57bad..b6502e1`. 17 of 18 tasks from the original implementation plan (`docs/superpowers/plans/2026-04-21-project-rag.md`) are complete; only the manual simulator smoke test is outstanding.

This phase verifies the feature end-to-end on a dedicated iOS simulator and closes any functional gaps that surface. It is explicitly NOT:

- A new feature build. Everything is in place already.
- A visual/UX polish pass. Functional verification only.
- A physical-device test. The glasses HUD side of SC #4 is deferred to hardware testing outside this session.

The phase produces: walkthrough evidence, a `GAPS.md` of findings, and either a clean verification or fixes committed to close blockers.
</domain>

<decisions>
## Implementation Decisions

### Verification Scope & Environment
- Use the iPhone 17 Pro simulator (`0D7C3AB2`) — the Helix-reserved slot per CLAUDE.md — booted as a dedicated instance, not shared.
- Debug build via `flutter run -d <sim-id>`; simulator cannot run release.
- Re-run `scripts/run_gate.sh` immediately before the walkthrough as a known-good baseline, so sim failures can be attributed cleanly to the walkthrough and not to a gate regression.
- For Success Criterion #4 (live conversation with citations), verify the **phone-side** behavior on the simulator: PROJECT CONTEXT injection, streamed answer references the project, citation chips render, tapping a chip reveals the excerpt. Full glasses-HUD verification is deferred to physical-device testing, captured as out-of-scope for this phase.

### Polish vs. Ship Threshold
- Failures are triaged per-finding into two buckets: **blocks-feature-use** (fix in phase) vs. **UX paper-cut** (capture as deferred todo in GAPS.md).
- Blocks-feature-use means: upload fails, retrieval returns no chunks when it should, citations don't appear, active-project persistence broken, crashes.
- Cost-conscious fixtures: 1 small TXT + 1 tiny 2-3 page PDF are sufficient per original plan Task 18. Embedding cost is negligible but the habit matters.
- No visual/UX polish in this phase — capture as todos for later, keep scope functional.

### Test Artifacts & Criteria
- Fixtures committed under `test/fixtures/project_rag/`:
  - `magic.txt` — 5-10 line handwritten snippet containing "The magic number is 42"
  - `sample.pdf` — synthetic 2-3 page PDF, generated once and checked in
- Canonical probe question: **"What is the magic number?"** Expected answer cites `[1]` and the cited source text contains "42".
- Evidence trail: screenshots at each critical transition (upload-ready, ask-answered, active-selected, chip-visible, delete-restore, after-restart) saved under `.planning/phases/01.1-project-rag-verification-polish-inserted/evidence/`.
- Deferred findings go into a structured `GAPS.md` in the phase directory with fields: severity (blocker/polish), file:line anchor, repro steps, recommended next action.

### Live-Session Phone-Side Verification Protocol
- Probe injection: prefer an existing debug seam (the widget test harness uses fake transcription via `test/helpers/test_helpers.dart` + `setupTestEngine`). For the sim walkthrough, use the HomeScreen's text-entry/voice-query path if available; if no release seam exists, add a one-line debug helper gated behind `kDebugMode`.
- Acceptance bar for SC #4 on phone: (a) active project set, (b) probe question arrives in transcript, (c) `PROJECT CONTEXT` block is injected into the system prompt (verify via debug log), (d) streamed answer references the TXT's unique fact, (e) citation chip renders beneath the answer, (f) tapping the chip reveals the excerpt.
- Distinguish RAG failures from pipeline failures: if citations fire correctly when an answer DOES stream, the RAG hook works and any answer-streaming failure is Phase 1/2 territory — record it against those phases, not 1.1. If citations don't fire given a good answer, it's a 1.1 bug.
- Include a regression check for the most-recent fix (`b6502e1` — clear stale citation chips at start of each response turn): ask two sequential probe questions in the same session, confirm the second turn's citations replace rather than accumulate.

### Claude's Discretion
- The exact debug seam used for probe injection (whether via voice mic in sim, a kDebugMode text-injection hook, or an existing HomeScreen entry field) is left to implementation-time discovery — whichever exists cleanly first.
- Screenshot filenames, GAPS.md column ordering, and evidence directory structure.
- Whether to run any of the existing Project RAG test suites (`test/services/projects/*`, `test/services/database/project_*`) as part of the pre-walkthrough confidence check in addition to the full gate.

Engine integration:
- `lib/services/conversation_engine.dart:154-159` — `_projectCitationsController` + `projectCitationsStream`
- `lib/services/conversation_engine.dart:2499-2509` — PROJECT CONTEXT injection hook + citation emission
- `lib/services/conversation_engine.dart:3484` — stream closed on disposal

Tests:
- 11 test files covering DB, services, DI widgets, and the engine hook — all passing in isolation per Task 17 run

### Established Patterns

- Singleton services with `.instance` getters and test-only `resetForTesting()` methods.
- All new persistence goes through Drift DAOs; FTS5 used for full-text search elsewhere in the codebase; this feature uses in-memory cosine over persisted vectors instead.
- StreamControllers for reactive UI (`projectCitationsStream` mirrors `aiResponseStream` pattern).
- API keys via `SettingsManager` + `FlutterSecureStorage`; OpenAI embeddings client reads the same key as the LLM provider.
- Test helpers in `test/helpers/test_helpers.dart` (`setupTestEngine` / `teardownTestEngine`) for engine tests.

### Integration Points

- App init in `lib/main.dart` — `ActiveProjectController.load()` before `runApp` so Home can synchronously read it; `ProjectsService.instance.purgeExpiredSoftDeletes()` kicked off unawaited after runtime boot.
- Live answer path — `ConversationEngine._generateResponse` prepends `PROJECT CONTEXT` when an active project is set and retrieval returns chunks; emits chunks to `projectCitationsStream` after.
- Home screen wires the active-project chip (app bar) and citation chips (below live answer) to controllers/streams.
